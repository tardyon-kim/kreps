# 회사 내부 Work OS 1차 설계

작성일: 2026-04-30

## 결정 요약

이 시스템은 처음부터 ERP나 그룹웨어 전체를 복제하지 않는다. 1차 목표는 회사의 모든 업무가 흩어지지 않도록 **전사 업무 허브형 Work OS**를 만드는 것이다.

가장 중요한 설계 원칙은 다음과 같다.

- 모든 업무는 하나의 공통 객체인 `업무 항목`으로 등록된다.
- 업무 항목에는 출처, 책임자, 담당자, 상태, 기한, 우선순위, 이력, 관련 문서가 남는다.
- 대표 지시, 관리자 요청, 직원 요청, 회의 후속 작업, 프로젝트 작업, 정기 업무를 같은 흐름에서 관리한다.
- 한국어와 영어를 포함한 다국어 사용을 1차 기반 요구사항으로 둔다.
- 테마는 개인별 사용성 요구와 장시간 업무 환경을 고려해 1차 기반 요구사항으로 둔다.
- 번역은 단순 기계번역 버튼이 아니라 회사 용어집, 업무 문맥, 원문 보존, 사람 검토를 포함한 번역 보조 구조로 설계한다.
- UI/UX는 업무용 제품에 맞게 조용하고 빠르며, 데이터 탐색과 상태 판단이 쉬운 디자인 시스템으로 설계한다.
- UI는 빠르면서도 세련되고 완성도 있게 보이도록, 절제된 색상과 명확한 타이포그래피, 섬세한 상태 표현을 사용한다.
- 운영 환경은 사내망 온프레미스를 기본으로 하고, 런타임 외부 네트워크 의존성을 최소화한다.
- AI 에이전트는 별도 허용된 통신 경로를 통해 사용할 수 있게 하되, 일반 애플리케이션 트래픽과 분리한다.
- 현재는 GitHub를 사용하지만, 추후 사내 Gitea로 이전될 수 있으므로 저장소와 CI/CD는 GitHub 전용 기능에 묶이지 않게 설계한다.
- 프로젝트, 결재, 문서, 일정, 보고, 자산 같은 기능은 업무 허브 위에 붙는 모듈로 확장한다.
- 일반 AI 에이전트는 1차 구현 범위에서 제외하되, 번역 보조 에이전트는 다국어 운영을 위한 우선 확장 후보로 별도 설계한다.

## 배경과 문제

현재 회사는 업무 체계가 거의 없고, 일이 대표 지시, 프로젝트 관리자 요청, 직원 요청 등 여러 경로로 발생한다. 이 상태에서 곧바로 프로젝트 관리, 결재, 문서관리, AI 에이전트를 각각 따로 만들면 시스템은 복잡해지지만 실제 업무 체계는 잡히지 않는다.

따라서 먼저 필요한 것은 기능의 종류가 아니라 **업무가 회사 안에서 어떻게 생기고, 누가 판단하고, 누가 처리하고, 언제 끝나는지**를 보이게 만드는 운영 체계다.

## 참고한 운영 원칙

조사한 기준은 다음과 같다.

- Kanban: 현재 일하는 방식에서 시작하고, 보이지 않는 업무를 시각화하며, 진행 중 업무를 제한하고 흐름을 관리한다.
- ISO 9001 프로세스 접근: 업무를 프로세스로 정의하고, Plan-Do-Check-Act 방식으로 점진적으로 개선한다.
- Jira Work Management: 반복 업무에 검토와 승인 단계를 둔 상태 기반 프로세스를 사용한다.
- OpenProject: 역할과 권한을 분리하고, 프로젝트 단위 권한을 세밀하게 제어한다.
- monday.com / Asana: 업무를 한곳에 모아 소유권, 상태, 기한, 차단 요소를 쉽게 보이게 한다.

이 시스템은 위 원칙을 그대로 복제하지 않고, 체계가 약한 중소 규모 회사에 맞게 얇고 실용적인 형태로 적용한다.

## 목표

1차 시스템의 목표는 다음과 같다.

- 업무가 메신저, 구두 지시, 개인 메모에 흩어지지 않게 한다.
- 모든 업무에 책임자, 담당자, 상태, 기한을 부여한다.
- 업무 발생부터 종료까지 이력을 남긴다.
- 직원, 조직, 권한을 업무 흐름과 연결한다.
- 한국어 사용자와 영어 사용자가 같은 업무 데이터를 함께 사용할 수 있게 한다.
- 직원이 별도 교육 없이도 내 업무, 업무 등록, 검색, 필터, 상태 변경을 빠르게 사용할 수 있게 한다.
- 반복되는 업무는 프로세스로 만들 수 있게 한다.
- 프로젝트성 업무와 단발성 업무를 같은 구조 안에서 관리한다.
- 나중에 AI 에이전트가 읽고 도울 수 있는 업무 데이터를 축적한다.

## 1차 제외 범위

다음은 1차 버전에서 제외한다.

- 회계, 급여, 세무, 재고, 생산 전체를 포괄하는 ERP 기능
- 복잡한 전자결재 엔진
- 모바일 앱
- 외부 SaaS 연동
- AI 에이전트 자동 실행
- 전사 모든 데이터의 실시간 자동 번역
- 마케팅 사이트식 랜딩 페이지나 장식 중심 UI
- 런타임 CDN 의존
- 배포 시 외부 패키지 저장소에 반드시 접속해야 하는 구조
- GitHub Actions에만 종속되는 배포 파이프라인
- 고급 리소스 배정, 예산 관리, 간트 차트
- 문서 버전관리 중심의 전문 문서관리 시스템

단, 이 기능들이 나중에 붙을 수 있도록 업무 항목, 권한, 파일, 이력 구조에는 확장 여지를 둔다.

## 전체 구조

시스템은 다음 계층으로 구성한다.

### 1. 조직 기반 계층

- 직원 계정
- 조직도
- 직책과 직무
- 역할
- 권한
- 조직장과 위임자

이 계층은 모든 기능의 접근 제어와 책임 범위 판단에 사용된다.

### 2. 업무 허브 계층

업무 허브는 시스템의 중심이다. 모든 업무는 `업무 항목`으로 등록된다.

업무 항목의 유형은 다음과 같이 시작한다.

- 지시
- 요청
- 프로젝트 작업
- 승인 필요 업무
- 정기 업무
- 이슈
- 회의 후속 작업

유형은 업무의 의미를 구분할 뿐, 기본 구조는 동일하다.

### 3. 업무 모듈 계층

업무 허브 위에 다음 모듈을 둔다.

- 프로젝트
- 승인/결재
- 문서/지식
- 일정
- 보고
- 자산/구매 요청

1차 구현에서는 프로젝트, 간단 승인, 문서/파일 첨부, 일정성 기한까지만 포함한다.

### 4. AI 확장 계층

AI 에이전트는 2차 이후에 붙인다. 1차에서는 AI가 사용할 수 있는 다음 기반만 남긴다.

- 업무 항목의 구조화된 상태
- 댓글과 변경 이력
- 첨부 파일 메타데이터
- 프로젝트와 업무의 관계
- 조직과 권한 범위
- 감사 가능한 액션 로그

다만 번역 보조는 일반 AI 에이전트와 분리해서 설계한다. 다국어 조직에서는 번역이 단순 편의 기능이 아니라 업무 이해의 기반이기 때문이다.

### 5. 국제화와 사용자 경험 계층

시스템 UI는 처음부터 다국어 구조로 만든다. 화면 문구, 메뉴, 버튼, 상태명, 오류 메시지는 코드에 직접 박지 않고 번역 리소스로 관리한다.

초기 지원 언어는 다음과 같다.

- 한국어
- 영어

사용자는 개인 설정에서 표시 언어와 테마를 선택할 수 있다. 조직 기본 언어도 별도로 둘 수 있다.

테마는 다음을 1차 기준으로 둔다.

- 시스템 기본값
- 라이트 모드
- 다크 모드

테마는 단순 색상 변경이 아니라 장시간 업무 사용을 고려한 대비, 테이블 가독성, 상태 색상, 경고 색상의 일관성을 포함한다.

### 6. UI/UX와 디자인 시스템 계층

UI/UX는 별도 장식보다 업무 속도와 신뢰성을 우선한다. 단, "조용하다"는 뜻은 밋밋하다는 뜻이 아니다. 화면은 정돈되고 세련되며, 매일 오래 봐도 피로하지 않은 완성도 있는 업무용 제품처럼 보여야 한다.

디자인 방향은 Ant Design, Carbon, Atlassian, Fluent 같은 업무용 디자인 시스템을 참고하되 그대로 복제하지 않고 회사 내부 업무에 맞게 단순화한다.

기본 레이아웃은 다음으로 둔다.

- 좌측 고정 사이드바: 내 업무, 업무 등록, 전사 업무, 프로젝트, 승인, 조직, 용어집, 설정
- 상단 작업 영역: 전역 검색, 빠른 업무 등록, 알림, 언어, 테마, 사용자 메뉴
- 중앙 업무 영역: 목록, 보드, 폼, 대시보드
- 우측 상세 패널: 선택한 업무의 상세, 댓글, 이력, 번역, 승인 액션

초기 화면은 `내 업무`를 기본 진입점으로 한다. 사용자가 로그인하면 오늘 처리할 업무, 검토 대기, 기한 임박, 번역 필요, 지연 위험을 먼저 본다.

디자인 시스템은 다음 토큰을 가진다.

- 색상 토큰
- 간격 토큰
- 타이포그래피 토큰
- 반경 토큰
- 그림자/경계 토큰
- 상태 토큰
- 테마 토큰

상태 색상은 의미 기반으로 관리한다.

- 정보: 진행 중, 배정됨
- 성공: 완료
- 경고: 기한 임박, 검토 필요
- 위험: 지연, 반려, 오류
- 중립: 초안, 보류, 취소

색상만으로 상태를 전달하지 않는다. 상태 배지에는 텍스트와 아이콘 또는 모양 차이를 함께 사용한다.

시각 스타일은 다음 기준을 따른다.

- 중립적인 배경과 선명한 본문 대비를 기본으로 한다.
- 브랜드 색상은 주요 액션과 선택 상태에 제한적으로 사용한다.
- 상태 색상은 화려함보다 의미 전달을 우선한다.
- 그림자는 적게 쓰고, 경계선과 레이어 색상으로 정보 구조를 나눈다.
- 아이콘은 일관된 두께와 크기로 사용한다.
- 업무 화면 안에 장식용 이미지, 배경 패턴, 과한 그라데이션을 넣지 않는다.
- 테이블과 폼은 밀도 있게 보이되 답답하지 않게 간격을 조절한다.
- 사용자가 오래 보는 화면은 라이트와 다크 모두 눈부심과 대비를 점검한다.

1차에서는 두 가지 화면 밀도를 지원할 수 있게 설계한다.

- 기본 밀도: 처음 사용하는 직원에게 편한 여백과 행 높이
- 컴팩트 밀도: 많은 업무를 빠르게 훑어보는 관리자용 밀도

1차 컴포넌트는 다음을 우선한다.

- 앱 셸
- 사이드바
- 상단바
- 버튼
- 아이콘 버튼
- 입력 필드
- 셀렉트
- 체크박스
- 토글
- 날짜 선택
- 상태 배지
- 사용자 아바타
- 데이터 테이블
- 필터 바
- 검색 입력
- 페이지네이션
- 탭
- 상세 패널
- 모달
- 토스트
- 빈 상태
- 로딩 스켈레톤
- 권한 없음 화면

컴포넌트는 한국어와 영어 문구 길이를 모두 고려한다. 버튼, 테이블 헤더, 상태 배지, 사이드바 항목은 긴 영어 문구 때문에 깨지지 않아야 한다. 불가피하게 줄임이 필요한 곳은 툴팁으로 전체 문구를 보여준다.

접근성 기준은 다음과 같다.

- 본문 텍스트 대비는 WCAG AA 수준을 만족한다.
- 키보드로 주요 업무 흐름을 사용할 수 있다.
- 포커스 링을 명확하게 보여준다.
- 아이콘 버튼에는 접근 가능한 이름을 제공한다.
- 오류는 색상만이 아니라 메시지와 위치로 알려준다.
- 로딩은 스피너만 쓰지 않고 스켈레톤과 명확한 상태 문구를 사용한다.

UX 품질 기준은 다음과 같다.

- 핵심 액션은 2클릭 이내로 접근한다.
- 업무 등록은 빠른 등록과 상세 등록을 분리한다.
- 목록은 기본적으로 검색, 필터, 정렬, 저장된 뷰를 제공한다.
- 상세 페이지로 이동하지 않아도 우측 패널에서 빠른 검토와 상태 변경이 가능하다.
- 위험한 액션은 확인 단계를 둔다.
- 저장, 실패, 권한 없음, 지연 상태를 즉시 피드백한다.
- 빈 상태는 다음 행동을 안내한다.

## 핵심 업무 흐름

기본 흐름은 다음과 같다.

`등록 -> 분류 -> 접수/승인 -> 배정 -> 진행 -> 검토 -> 종료`

각 단계의 의미는 다음과 같다.

- 등록: 누구든 권한 범위 안에서 업무를 생성하거나 요청한다.
- 분류: 업무 유형, 우선순위, 관련 조직, 프로젝트 여부를 정한다.
- 접수/승인: 진행할 업무인지, 보류 또는 반려할 업무인지 판단한다.
- 배정: 책임자와 담당자를 정하고 기한을 부여한다.
- 진행: 담당자가 처리하고 댓글, 파일, 상태를 갱신한다.
- 검토: 요청자, 책임자, 승인자가 결과를 확인한다.
- 종료: 완료, 반려, 취소, 보류 중 하나로 닫는다.

이 흐름은 모든 업무에 강제로 복잡하게 적용하지 않는다. 단순 업무는 `등록 -> 배정 -> 진행 -> 종료`로 빠르게 처리할 수 있게 하고, 승인이나 검토가 필요한 업무만 전체 단계를 사용한다.

## 업무 항목 설계

업무 항목은 다음 필드를 가진다.

- 제목
- 설명
- 유형
- 출처
- 상태
- 우선순위
- 요청자
- 책임자
- 담당자
- 관련 조직
- 관련 프로젝트
- 시작일
- 기한
- 완료일
- 공개 범위
- 첨부 파일
- 댓글
- 체크리스트
- 변경 이력
- 태그
- 원문 언어
- 번역 상태
- 번역본

업무 항목은 하위 업무를 가질 수 있다. 하위 업무는 독립된 담당자와 상태를 가질 수 있지만, 상위 업무가 종료될 때 완료 여부를 함께 확인한다.

사용자가 작성한 제목, 설명, 댓글, 완료 보고는 원문을 반드시 보존한다. 번역본은 원문을 대체하지 않고 별도 데이터로 저장한다.

## 상태 모델

초기 상태는 다음으로 둔다.

- 초안
- 등록됨
- 검토 대기
- 승인됨
- 배정됨
- 진행 중
- 보류
- 완료 보고
- 반려
- 완료
- 취소

사용자 화면에는 너무 많은 상태를 한 번에 노출하지 않는다. 기본 보드는 다음 그룹으로 묶어 보여준다.

- 대기
- 진행
- 검토
- 완료
- 보류/반려

내부 상태는 세밀하게 남기되, 사용자가 보는 화면은 단순하게 유지한다.

## 권한 모델

권한은 `Role + Permission + Scope` 구조로 설계한다.

- Role: 권한 묶음
- Permission: 실제 행동 권한
- Scope: 권한이 적용되는 범위

초기 역할은 다음과 같다.

- 시스템 관리자
- 조직 관리자
- 업무 관리자
- 프로젝트 관리자
- 일반 직원
- 조회자

초기 Permission 예시는 다음과 같다.

- 직원 관리
- 조직 관리
- 역할 관리
- 업무 등록
- 업무 승인
- 업무 배정
- 업무 상태 변경
- 업무 종료
- 프로젝트 생성
- 프로젝트 구성원 관리
- 파일 첨부
- 댓글 작성
- 보고서 조회
- 감사 이력 조회

Scope는 다음 단위로 둔다.

- 전체
- 조직 및 하위 조직
- 프로젝트
- 본인 관련 업무

이 구조를 사용하면 처음에는 표준 역할만 제공하면서도, 나중에 역할과 권한을 화면에서 직접 편집하는 방식으로 확장할 수 있다.

## 프로젝트 모듈

프로젝트는 큰 목표나 기간이 있는 업무 묶음이다. 프로젝트 자체도 업무 항목과 연결된다.

프로젝트의 기본 정보는 다음과 같다.

- 프로젝트명
- 목적
- 상태
- 기간
- 프로젝트 관리자
- 참여 조직
- 참여자
- 마일스톤
- 연결 업무 항목
- 파일
- 댓글

프로젝트 화면은 다음 뷰를 제공한다.

- 개요
- 업무 목록
- 상태 보드
- 마일스톤
- 파일
- 활동 이력

1차에서는 목록과 상태 보드를 우선하고, 마일스톤은 단순한 날짜 기반으로 둔다.

## 승인/결재 모듈

1차 승인 기능은 전자결재 전체가 아니라 업무 항목의 상태 전환으로 구현한다.

예시는 다음과 같다.

- 업무 진행 승인
- 완료 검토
- 구매 요청 승인
- 휴가 요청 승인
- 문서 검토 승인

승인선은 처음부터 복잡한 템플릿 엔진으로 만들지 않는다. 1차에서는 다음 방식으로 충분하다.

- 업무 유형별 기본 승인자
- 조직장 승인
- 프로젝트 관리자 승인
- 수동 승인자 지정

추후에는 다단계 승인선, 대결, 전결, 병렬 승인으로 확장한다.

## 문서/파일

1차에서는 별도 문서관리 시스템보다 업무와 프로젝트에 파일을 붙이는 방식을 우선한다.

파일은 다음 정보와 함께 저장한다.

- 파일명
- 크기
- MIME 타입
- 업로드한 사용자
- 업로드 시각
- 연결 대상
- 접근 범위

문서 본문 관리, 버전 비교, 문서 번호 체계는 후속 단계로 둔다.

## 다국어와 번역

다국어 지원은 두 층으로 나눈다.

### UI 국제화

UI 국제화는 1차 필수 범위다.

- 모든 메뉴, 버튼, 상태, 오류 메시지는 번역 키로 관리한다.
- 기본 언어는 사용자 설정을 우선한다.
- 사용자 설정이 없으면 조직 기본 언어를 따른다.
- 조직 기본 언어도 없으면 한국어를 기본값으로 둔다.
- 날짜, 시간, 숫자 표기는 사용자 언어와 지역 설정을 반영한다.

### 업무 콘텐츠 번역

업무 콘텐츠 번역은 원문 보존을 기본 원칙으로 한다. 제목, 설명, 댓글, 완료 보고, 프로젝트 설명은 사용자가 작성한 원문을 그대로 저장하고, 번역본은 별도 필드 또는 별도 테이블에 저장한다.

번역 상태는 다음으로 둔다.

- 번역 없음
- 자동 번역됨
- 검토 필요
- 검토 완료
- 원문 수정으로 재번역 필요

번역 표시 방식은 다음과 같다.

- 사용자의 표시 언어와 원문 언어가 다르면 번역 보기 버튼을 제공한다.
- 자동 번역 결과에는 자동 번역임을 표시한다.
- 중요한 업무 항목은 사용자가 번역을 검토 완료로 표시할 수 있다.
- 원문과 번역본을 함께 볼 수 있게 한다.

### 회사 용어집

일반 번역 품질을 보완하기 위해 회사 용어집을 둔다.

용어집 항목은 다음 정보를 가진다.

- 원어
- 한국어 표현
- 영어 표현
- 설명
- 사용 예시
- 적용 범위
- 마지막 수정자
- 마지막 수정일

번역 보조 기능은 용어집을 우선 반영한다. 예를 들어 제품명, 고객명, 부서명, 직책, 내부 약어는 일반 번역기가 임의로 바꾸지 않도록 한다.

### 번역 보조 에이전트

번역 보조 에이전트는 전체 AI 에이전트보다 먼저 붙일 수 있는 제한적 AI 기능이다. 목적은 사용자를 대신해 업무를 처리하는 것이 아니라, 다국어 협업에서 의미가 어긋나지 않도록 돕는 것이다.

번역 보조 에이전트의 역할은 다음과 같다.

- 업무 제목과 설명 번역
- 댓글 번역
- 완료 보고 번역
- 프로젝트 설명 번역
- 회사 용어집 기반 표현 통일
- 모호한 표현 표시
- 번역 품질 검토 요청 표시

번역 보조 에이전트는 다음 제한을 가진다.

- 원문을 수정하지 않는다.
- 업무 상태를 변경하지 않는다.
- 승인, 종료, 담당자 변경 같은 업무 액션을 수행하지 않는다.
- 사용자가 접근 권한을 가진 업무만 번역한다.
- 번역 결과에는 자동 생성 표시를 남긴다.

1차 구현에서는 번역 데이터 모델, UI 진입점, 용어집, 수동 검토 흐름을 만든다. 실제 AI 번역 엔진은 어댑터 구조로 분리해 내부망 정책과 모델 선택이 정해졌을 때 연결한다.

## 알림과 대시보드

초기 알림 대상은 다음과 같다.

- 내게 배정된 업무
- 내 요청 업무의 상태 변경
- 승인 요청
- 기한 임박
- 기한 초과
- 댓글 언급
- 프로젝트 변경

대시보드는 역할별로 다르게 보여준다.

- 일반 직원: 내 요청, 내 업무, 승인 대기, 기한 임박
- 업무 관리자: 미분류 업무, 지연 업무, 담당자 없는 업무
- 프로젝트 관리자: 프로젝트별 진행률, 차단 업무, 마일스톤 위험
- 경영진: 전사 업무량, 지연 현황, 프로젝트 현황, 조직별 병목

## AI 에이전트 확장 방향

AI는 1차 구현에서 제외하지만 다음 순서로 붙이는 것이 적절하다.

### 1단계: 읽기 전용 보조

- 번역 보조
- 업무 요약
- 프로젝트 진행 요약
- 지연 업무 설명
- 회의록에서 할 일 후보 추출
- 사내 자료 검색

### 2단계: 초안 작성

- 업무 설명 초안
- 보고서 초안
- 회의록 정리
- 완료 보고 초안
- 프로젝트 주간 보고 초안

### 3단계: 추천

- 담당자 후보 추천
- 우선순위 추천
- 유사 업무 추천
- 누락된 체크리스트 제안
- 지연 위험 감지

### 4단계: 제한적 자동화

- 정기 보고 생성
- 반복 업무 생성
- 승인자에게 리마인드
- 프로젝트 위험 알림

AI는 항상 사용자 권한 범위 안의 데이터만 읽어야 한다. AI가 업무 생성, 상태 변경, 승인, 종료 같은 실제 액션을 수행할 때는 사람의 확인을 받는 구조로 둔다.

## 배포와 보안

시스템은 회사 내부망 전용 온프레미스 시스템으로 운영한다. 외부망에서 직접 접근할 수 없어야 하며, 일반 애플리케이션 기능은 런타임에 외부 인터넷을 필요로 하지 않아야 한다.

네트워크 전제는 다음과 같다.

- 사용자는 사내망 또는 허용된 VPN에서만 접속한다.
- 외부에서 Web/API 서버로 들어오는 인바운드 접근은 차단한다.
- 앱 화면은 CDN, 외부 폰트, 외부 아이콘, 외부 스크립트에 의존하지 않는다.
- 컨테이너 이미지는 운영 전에 사내 서버 또는 사내 레지스트리에 준비한다.
- 패키지 설치와 빌드는 개발/빌드 환경에서 수행하고, 운영 서버는 빌드 산출물을 실행하는 역할을 우선한다.
- 보안 패치와 버전 업그레이드는 외부망에서 검증한 뒤 사내망으로 반입하는 절차를 둔다.

AI 에이전트 통신은 일반 네트워크 정책과 분리한다. 에이전트 사용을 위해 이미 허용된 경로가 있더라도, 애플리케이션 전체가 자유롭게 외부망에 접근하는 구조로 만들지 않는다.

AI 통신 원칙은 다음과 같다.

- AI 호출은 `AI Gateway` 또는 `Agent Adapter`를 통해서만 수행한다.
- AI Gateway는 허용된 모델 엔드포인트, 인증 정보, 요청 제한, 감사 로그를 중앙에서 관리한다.
- 업무 데이터는 사용자의 권한 범위 안에서만 AI로 전달한다.
- 번역, 요약, 추천 결과에는 AI 생성 표시와 요청 이력을 남긴다.
- AI 기능이 실패해도 핵심 업무 등록, 조회, 상태 변경은 계속 동작해야 한다.

권장 배포 구조는 다음과 같다.

- Docker Compose 기반
- Web/API 컨테이너
- PostgreSQL
- 파일 저장 볼륨 또는 NAS 마운트
- 내부망 reverse proxy
- 백업 컨테이너 또는 백업 스크립트
- 선택 사항: 사내 컨테이너 레지스트리
- 선택 사항: AI Gateway 컨테이너

배포는 최대한 단순해야 한다. 1차 운영자는 다음 흐름으로 설치할 수 있어야 한다.

1. 릴리스 패키지를 서버에 복사한다.
2. `.env` 또는 설치 설정 파일을 작성한다.
3. `docker compose up -d` 또는 제공된 설치 스크립트를 실행한다.
4. 초기 관리자 계정을 만든다.
5. 상태 점검 화면에서 DB, 파일 저장소, 백업 경로, AI Gateway 연결 상태를 확인한다.

릴리스 패키지는 다음을 포함한다.

- `compose.yml`
- `.env.example`
- 설치 스크립트
- 업그레이드 스크립트
- 백업 스크립트
- 복원 스크립트
- 헬스체크 스크립트
- DB 마이그레이션
- 운영 가이드

오프라인 또는 제한망 배포를 위해 다음을 지원한다.

- 컨테이너 이미지 tar 내보내기와 가져오기
- npm/pnpm 패키지 락파일 고정
- 빌드 산출물에 정적 자산 포함
- 외부 폰트 대신 로컬 폰트 또는 시스템 폰트 사용
- 외부 API 미연결 상태에서도 앱 부팅 가능
- AI 기능 비활성화 모드

1차 보안 기준은 다음과 같다.

- 내부망 접속만 허용
- 비밀번호 해시 저장
- 세션 만료
- 계정 비활성화
- 역할 기반 접근 제어
- 업무 상태 변경 이력
- 관리자 감사 로그
- 환경변수와 시크릿 분리
- 운영 로그와 감사 로그 분리
- 백업 파일 접근 권한 제한

2차 이후에는 비밀번호 정책, 로그인 실패 잠금, 2단계 인증, IP 제한, 중요 작업 재인증을 검토한다.

## 저장소와 배포 파이프라인

현재 개발 저장소는 GitHub를 사용한다. 하지만 추후 사내망 Gitea로 이전될 가능성이 높으므로 저장소 운영은 Git 호스팅 독립적으로 설계한다.

원칙은 다음과 같다.

- GitHub 전용 기능에 핵심 개발 흐름을 묶지 않는다.
- 빌드, 테스트, 린트, 마이그레이션, 배포 명령은 로컬 스크립트로 제공한다.
- CI는 GitHub Actions와 Gitea Actions 또는 사내 CI가 같은 스크립트를 호출하게 한다.
- 이슈/PR 템플릿은 GitHub와 Gitea에서 모두 읽을 수 있는 Markdown으로 관리한다.
- 배포는 GitHub 릴리스에 의존하지 않고, git tag와 빌드 산출물로 재현 가능하게 한다.
- 사내 Gitea 이전 시 Git remote URL 변경만으로 개발 흐름을 유지할 수 있게 한다.

Gitea 이전 방식은 다음 순서를 권장한다.

1. 사내 Gitea에 빈 저장소를 만든다.
2. GitHub 저장소를 Gitea로 미러링하거나 로컬에서 새 remote를 추가한다.
3. 기본 브랜치, 태그, 이슈 템플릿, 문서가 정상 이전됐는지 확인한다.
4. CI/CD 변수를 Gitea 또는 사내 CI에 다시 등록한다.
5. 개발자 로컬 remote를 Gitea로 전환한다.
6. GitHub는 필요 시 읽기 전용 백업 또는 외부 협업용 미러로 유지한다.

## 성공 기준

1차 시스템이 성공했는지는 다음으로 판단한다.

- 주요 업무가 메신저나 구두 지시가 아니라 시스템에 등록된다.
- 담당자 없는 업무가 줄어든다.
- 기한 초과 업무가 보인다.
- 대표와 관리자가 업무 현황을 따로 물어보지 않아도 확인할 수 있다.
- 직원이 “내가 지금 해야 할 일”을 한 화면에서 볼 수 있다.
- 한국어 사용자와 영어 사용자가 같은 업무 항목을 각자 이해 가능한 언어로 확인할 수 있다.
- 사용자가 검색, 필터, 빠른 등록, 상태 변경을 교육 없이 사용할 수 있다.
- 사내 서버에서 문서화된 절차로 설치, 업그레이드, 백업, 복원이 가능하다.
- 외부 인터넷이 없어도 핵심 업무 기능이 동작한다.
- 완료와 반려의 근거가 이력으로 남는다.
- 반복 업무가 점차 템플릿화된다.

## 주요 위험과 대응

### 위험: 시스템이 너무 무거워져서 직원이 쓰지 않는다

대응: 처음 화면은 `내 업무`, `업무 등록`, `업무 목록`, `프로젝트` 중심으로 단순하게 만든다. 승인과 분류는 필요한 업무에만 적용한다.

### 위험: 대표 지시가 여전히 구두나 메신저에 남는다

대응: 대표나 비서, 업무 관리자가 빠르게 지시를 대신 등록할 수 있는 `빠른 등록` 기능을 둔다.

### 위험: 권한이 복잡해져 운영이 어렵다

대응: 내부 구조는 Role + Permission + Scope로 만들되, 1차 화면은 표준 역할만 제공한다.

### 위험: AI를 나중에 붙이기 어렵다

대응: 업무 항목, 댓글, 파일, 이력, 권한을 구조화해 저장한다. AI 전용 로그와 권한 검사를 후속 단계에서 추가할 수 있게 한다.

### 위험: 번역 때문에 업무 의미가 왜곡된다

대응: 원문을 항상 보존하고, 자동 번역 표시와 검토 상태를 둔다. 회사 용어집을 사용해 제품명, 고객명, 부서명, 내부 약어가 임의로 번역되지 않게 한다.

### 위험: 언어와 테마를 나중에 붙이기 어렵다

대응: 1차부터 UI 문구를 번역 리소스로 분리하고, 색상은 디자인 토큰으로 관리한다. 화면 컴포넌트는 긴 영어 문구와 한국어 문구가 모두 깨지지 않게 설계한다.

### 위험: UI가 보기에는 예쁘지만 실제 업무에는 느리다

대응: 장식보다 목록, 필터, 검색, 빠른 등록, 상세 패널, 저장된 뷰를 우선한다. 모든 주요 화면은 "오늘 내가 해야 할 일"과 "관리자가 병목을 찾는 일"을 기준으로 검증한다.

### 위험: 디자인이 화면마다 달라져 유지보수가 어려워진다

대응: 1차부터 디자인 토큰과 공통 컴포넌트를 정의한다. 새 화면은 토큰과 컴포넌트만 사용하고, 임의 색상과 임의 간격을 금지한다.

### 위험: 제한망에서 배포가 실패한다

대응: 운영 서버가 외부 패키지 저장소나 CDN에 접근해야만 동작하는 구조를 금지한다. 컨테이너 이미지, 정적 자산, 마이그레이션, 스크립트를 릴리스 패키지에 포함하고, 오프라인 설치 절차를 문서화한다.

### 위험: AI 통신 경로가 일반 앱 보안 경계를 흐린다

대응: AI 호출은 AI Gateway 또는 Agent Adapter로 격리한다. 일반 Web/API 서버가 임의 외부 URL로 나가지 않게 하고, AI 요청은 권한 검사, 요청 로그, 실패 시 graceful fallback을 가진다.

### 위험: GitHub에서 Gitea로 이전할 때 개발 흐름이 깨진다

대응: GitHub Actions에만 의존하지 않고, 모든 빌드/테스트/배포를 로컬 스크립트로 제공한다. CI는 이 스크립트를 호출하는 얇은 래퍼로 유지한다.

## 최종 판단

가장 적절한 1차 방향은 **전사 업무 허브형 Work OS**다.

이 방향은 현재 체계가 약한 회사에 맞다. ERP처럼 무겁지 않고, 단순 프로젝트 관리툴처럼 좁지도 않다. 회사의 일을 먼저 한곳에 모으고, 상태와 책임을 보이게 만든 뒤, 프로젝트, 결재, 문서, AI 에이전트를 단계적으로 붙일 수 있다.
