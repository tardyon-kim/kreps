# 회사 내부 Work OS 1차 설계

작성일: 2026-04-30

## 결정 요약

이 시스템은 처음부터 ERP나 그룹웨어 전체를 복제하지 않는다. 1차 목표는 회사의 모든 업무가 흩어지지 않도록 **전사 업무 허브형 Work OS**를 만드는 것이다.

가장 중요한 설계 원칙은 다음과 같다.

- 모든 업무는 하나의 공통 객체인 `업무 항목`으로 등록된다.
- 업무 항목에는 출처, 책임자, 담당자, 상태, 기한, 우선순위, 이력, 관련 문서가 남는다.
- 대표 지시, 관리자 요청, 직원 요청, 회의 후속 작업, 프로젝트 작업, 정기 업무를 같은 흐름에서 관리한다.
- 한국어와 영어를 포함한 다국어 사용을 1차 기반 요구사항으로 둔다.
- 테마는 개인별 사용성 요구와 장시간 업무 환경을 고려해 1차 기반 요구사항으로 둔다.
- 번역은 단순 기계번역 버튼이 아니라 회사 용어집, 업무 문맥, 원문 보존, 사람 검토를 포함한 번역 보조 구조로 설계한다.
- 프로젝트, 결재, 문서, 일정, 보고, 자산 같은 기능은 업무 허브 위에 붙는 모듈로 확장한다.
- 일반 AI 에이전트는 1차 구현 범위에서 제외하되, 번역 보조 에이전트는 다국어 운영을 위한 우선 확장 후보로 별도 설계한다.

## 배경과 문제

현재 회사는 업무 체계가 거의 없고, 일이 대표 지시, 프로젝트 관리자 요청, 직원 요청 등 여러 경로로 발생한다. 이 상태에서 곧바로 프로젝트 관리, 결재, 문서관리, AI 에이전트를 각각 따로 만들면 시스템은 복잡해지지만 실제 업무 체계는 잡히지 않는다.

따라서 먼저 필요한 것은 기능의 종류가 아니라 **업무가 회사 안에서 어떻게 생기고, 누가 판단하고, 누가 처리하고, 언제 끝나는지**를 보이게 만드는 운영 체계다.

## 참고한 운영 원칙

조사한 기준은 다음과 같다.

- Kanban: 현재 일하는 방식에서 시작하고, 보이지 않는 업무를 시각화하며, 진행 중 업무를 제한하고 흐름을 관리한다.
- ISO 9001 프로세스 접근: 업무를 프로세스로 정의하고, Plan-Do-Check-Act 방식으로 점진적으로 개선한다.
- Jira Work Management: 반복 업무에 검토와 승인 단계를 둔 상태 기반 프로세스를 사용한다.
- OpenProject: 역할과 권한을 분리하고, 프로젝트 단위 권한을 세밀하게 제어한다.
- monday.com / Asana: 업무를 한곳에 모아 소유권, 상태, 기한, 차단 요소를 쉽게 보이게 한다.

이 시스템은 위 원칙을 그대로 복제하지 않고, 체계가 약한 중소 규모 회사에 맞게 얇고 실용적인 형태로 적용한다.

## 목표

1차 시스템의 목표는 다음과 같다.

- 업무가 메신저, 구두 지시, 개인 메모에 흩어지지 않게 한다.
- 모든 업무에 책임자, 담당자, 상태, 기한을 부여한다.
- 업무 발생부터 종료까지 이력을 남긴다.
- 직원, 조직, 권한을 업무 흐름과 연결한다.
- 한국어 사용자와 영어 사용자가 같은 업무 데이터를 함께 사용할 수 있게 한다.
- 반복되는 업무는 프로세스로 만들 수 있게 한다.
- 프로젝트성 업무와 단발성 업무를 같은 구조 안에서 관리한다.
- 나중에 AI 에이전트가 읽고 도울 수 있는 업무 데이터를 축적한다.

## 1차 제외 범위

다음은 1차 버전에서 제외한다.

- 회계, 급여, 세무, 재고, 생산 전체를 포괄하는 ERP 기능
- 복잡한 전자결재 엔진
- 모바일 앱
- 외부 SaaS 연동
- AI 에이전트 자동 실행
- 전사 모든 데이터의 실시간 자동 번역
- 고급 리소스 배정, 예산 관리, 간트 차트
- 문서 버전관리 중심의 전문 문서관리 시스템

단, 이 기능들이 나중에 붙을 수 있도록 업무 항목, 권한, 파일, 이력 구조에는 확장 여지를 둔다.

## 전체 구조

시스템은 다음 계층으로 구성한다.

### 1. 조직 기반 계층

- 직원 계정
- 조직도
- 직책과 직무
- 역할
- 권한
- 조직장과 위임자

이 계층은 모든 기능의 접근 제어와 책임 범위 판단에 사용된다.

### 2. 업무 허브 계층

업무 허브는 시스템의 중심이다. 모든 업무는 `업무 항목`으로 등록된다.

업무 항목의 유형은 다음과 같이 시작한다.

- 지시
- 요청
- 프로젝트 작업
- 승인 필요 업무
- 정기 업무
- 이슈
- 회의 후속 작업

유형은 업무의 의미를 구분할 뿐, 기본 구조는 동일하다.

### 3. 업무 모듈 계층

업무 허브 위에 다음 모듈을 둔다.

- 프로젝트
- 승인/결재
- 문서/지식
- 일정
- 보고
- 자산/구매 요청

1차 구현에서는 프로젝트, 간단 승인, 문서/파일 첨부, 일정성 기한까지만 포함한다.

### 4. AI 확장 계층

AI 에이전트는 2차 이후에 붙인다. 1차에서는 AI가 사용할 수 있는 다음 기반만 남긴다.

- 업무 항목의 구조화된 상태
- 댓글과 변경 이력
- 첨부 파일 메타데이터
- 프로젝트와 업무의 관계
- 조직과 권한 범위
- 감사 가능한 액션 로그

다만 번역 보조는 일반 AI 에이전트와 분리해서 설계한다. 다국어 조직에서는 번역이 단순 편의 기능이 아니라 업무 이해의 기반이기 때문이다.

### 5. 국제화와 사용자 경험 계층

시스템 UI는 처음부터 다국어 구조로 만든다. 화면 문구, 메뉴, 버튼, 상태명, 오류 메시지는 코드에 직접 박지 않고 번역 리소스로 관리한다.

초기 지원 언어는 다음과 같다.

- 한국어
- 영어

사용자는 개인 설정에서 표시 언어와 테마를 선택할 수 있다. 조직 기본 언어도 별도로 둘 수 있다.

테마는 다음을 1차 기준으로 둔다.

- 시스템 기본값
- 라이트 모드
- 다크 모드

테마는 단순 색상 변경이 아니라 장시간 업무 사용을 고려한 대비, 테이블 가독성, 상태 색상, 경고 색상의 일관성을 포함한다.

## 핵심 업무 흐름

기본 흐름은 다음과 같다.

`등록 -> 분류 -> 접수/승인 -> 배정 -> 진행 -> 검토 -> 종료`

각 단계의 의미는 다음과 같다.

- 등록: 누구든 권한 범위 안에서 업무를 생성하거나 요청한다.
- 분류: 업무 유형, 우선순위, 관련 조직, 프로젝트 여부를 정한다.
- 접수/승인: 진행할 업무인지, 보류 또는 반려할 업무인지 판단한다.
- 배정: 책임자와 담당자를 정하고 기한을 부여한다.
- 진행: 담당자가 처리하고 댓글, 파일, 상태를 갱신한다.
- 검토: 요청자, 책임자, 승인자가 결과를 확인한다.
- 종료: 완료, 반려, 취소, 보류 중 하나로 닫는다.

이 흐름은 모든 업무에 강제로 복잡하게 적용하지 않는다. 단순 업무는 `등록 -> 배정 -> 진행 -> 종료`로 빠르게 처리할 수 있게 하고, 승인이나 검토가 필요한 업무만 전체 단계를 사용한다.

## 업무 항목 설계

업무 항목은 다음 필드를 가진다.

- 제목
- 설명
- 유형
- 출처
- 상태
- 우선순위
- 요청자
- 책임자
- 담당자
- 관련 조직
- 관련 프로젝트
- 시작일
- 기한
- 완료일
- 공개 범위
- 첨부 파일
- 댓글
- 체크리스트
- 변경 이력
- 태그
- 원문 언어
- 번역 상태
- 번역본

업무 항목은 하위 업무를 가질 수 있다. 하위 업무는 독립된 담당자와 상태를 가질 수 있지만, 상위 업무가 종료될 때 완료 여부를 함께 확인한다.

사용자가 작성한 제목, 설명, 댓글, 완료 보고는 원문을 반드시 보존한다. 번역본은 원문을 대체하지 않고 별도 데이터로 저장한다.

## 상태 모델

초기 상태는 다음으로 둔다.

- 초안
- 등록됨
- 검토 대기
- 승인됨
- 배정됨
- 진행 중
- 보류
- 완료 보고
- 반려
- 완료
- 취소

사용자 화면에는 너무 많은 상태를 한 번에 노출하지 않는다. 기본 보드는 다음 그룹으로 묶어 보여준다.

- 대기
- 진행
- 검토
- 완료
- 보류/반려

내부 상태는 세밀하게 남기되, 사용자가 보는 화면은 단순하게 유지한다.

## 권한 모델

권한은 `Role + Permission + Scope` 구조로 설계한다.

- Role: 권한 묶음
- Permission: 실제 행동 권한
- Scope: 권한이 적용되는 범위

초기 역할은 다음과 같다.

- 시스템 관리자
- 조직 관리자
- 업무 관리자
- 프로젝트 관리자
- 일반 직원
- 조회자

초기 Permission 예시는 다음과 같다.

- 직원 관리
- 조직 관리
- 역할 관리
- 업무 등록
- 업무 승인
- 업무 배정
- 업무 상태 변경
- 업무 종료
- 프로젝트 생성
- 프로젝트 구성원 관리
- 파일 첨부
- 댓글 작성
- 보고서 조회
- 감사 이력 조회

Scope는 다음 단위로 둔다.

- 전체
- 조직 및 하위 조직
- 프로젝트
- 본인 관련 업무

이 구조를 사용하면 처음에는 표준 역할만 제공하면서도, 나중에 역할과 권한을 화면에서 직접 편집하는 방식으로 확장할 수 있다.

## 프로젝트 모듈

프로젝트는 큰 목표나 기간이 있는 업무 묶음이다. 프로젝트 자체도 업무 항목과 연결된다.

프로젝트의 기본 정보는 다음과 같다.

- 프로젝트명
- 목적
- 상태
- 기간
- 프로젝트 관리자
- 참여 조직
- 참여자
- 마일스톤
- 연결 업무 항목
- 파일
- 댓글

프로젝트 화면은 다음 뷰를 제공한다.

- 개요
- 업무 목록
- 상태 보드
- 마일스톤
- 파일
- 활동 이력

1차에서는 목록과 상태 보드를 우선하고, 마일스톤은 단순한 날짜 기반으로 둔다.

## 승인/결재 모듈

1차 승인 기능은 전자결재 전체가 아니라 업무 항목의 상태 전환으로 구현한다.

예시는 다음과 같다.

- 업무 진행 승인
- 완료 검토
- 구매 요청 승인
- 휴가 요청 승인
- 문서 검토 승인

승인선은 처음부터 복잡한 템플릿 엔진으로 만들지 않는다. 1차에서는 다음 방식으로 충분하다.

- 업무 유형별 기본 승인자
- 조직장 승인
- 프로젝트 관리자 승인
- 수동 승인자 지정

추후에는 다단계 승인선, 대결, 전결, 병렬 승인으로 확장한다.

## 문서/파일

1차에서는 별도 문서관리 시스템보다 업무와 프로젝트에 파일을 붙이는 방식을 우선한다.

파일은 다음 정보와 함께 저장한다.

- 파일명
- 크기
- MIME 타입
- 업로드한 사용자
- 업로드 시각
- 연결 대상
- 접근 범위

문서 본문 관리, 버전 비교, 문서 번호 체계는 후속 단계로 둔다.

## 다국어와 번역

다국어 지원은 두 층으로 나눈다.

### UI 국제화

UI 국제화는 1차 필수 범위다.

- 모든 메뉴, 버튼, 상태, 오류 메시지는 번역 키로 관리한다.
- 기본 언어는 사용자 설정을 우선한다.
- 사용자 설정이 없으면 조직 기본 언어를 따른다.
- 조직 기본 언어도 없으면 한국어를 기본값으로 둔다.
- 날짜, 시간, 숫자 표기는 사용자 언어와 지역 설정을 반영한다.

### 업무 콘텐츠 번역

업무 콘텐츠 번역은 원문 보존을 기본 원칙으로 한다. 제목, 설명, 댓글, 완료 보고, 프로젝트 설명은 사용자가 작성한 원문을 그대로 저장하고, 번역본은 별도 필드 또는 별도 테이블에 저장한다.

번역 상태는 다음으로 둔다.

- 번역 없음
- 자동 번역됨
- 검토 필요
- 검토 완료
- 원문 수정으로 재번역 필요

번역 표시 방식은 다음과 같다.

- 사용자의 표시 언어와 원문 언어가 다르면 번역 보기 버튼을 제공한다.
- 자동 번역 결과에는 자동 번역임을 표시한다.
- 중요한 업무 항목은 사용자가 번역을 검토 완료로 표시할 수 있다.
- 원문과 번역본을 함께 볼 수 있게 한다.

### 회사 용어집

일반 번역 품질을 보완하기 위해 회사 용어집을 둔다.

용어집 항목은 다음 정보를 가진다.

- 원어
- 한국어 표현
- 영어 표현
- 설명
- 사용 예시
- 적용 범위
- 마지막 수정자
- 마지막 수정일

번역 보조 기능은 용어집을 우선 반영한다. 예를 들어 제품명, 고객명, 부서명, 직책, 내부 약어는 일반 번역기가 임의로 바꾸지 않도록 한다.

### 번역 보조 에이전트

번역 보조 에이전트는 전체 AI 에이전트보다 먼저 붙일 수 있는 제한적 AI 기능이다. 목적은 사용자를 대신해 업무를 처리하는 것이 아니라, 다국어 협업에서 의미가 어긋나지 않도록 돕는 것이다.

번역 보조 에이전트의 역할은 다음과 같다.

- 업무 제목과 설명 번역
- 댓글 번역
- 완료 보고 번역
- 프로젝트 설명 번역
- 회사 용어집 기반 표현 통일
- 모호한 표현 표시
- 번역 품질 검토 요청 표시

번역 보조 에이전트는 다음 제한을 가진다.

- 원문을 수정하지 않는다.
- 업무 상태를 변경하지 않는다.
- 승인, 종료, 담당자 변경 같은 업무 액션을 수행하지 않는다.
- 사용자가 접근 권한을 가진 업무만 번역한다.
- 번역 결과에는 자동 생성 표시를 남긴다.

1차 구현에서는 번역 데이터 모델, UI 진입점, 용어집, 수동 검토 흐름을 만든다. 실제 AI 번역 엔진은 어댑터 구조로 분리해 내부망 정책과 모델 선택이 정해졌을 때 연결한다.

## 알림과 대시보드

초기 알림 대상은 다음과 같다.

- 내게 배정된 업무
- 내 요청 업무의 상태 변경
- 승인 요청
- 기한 임박
- 기한 초과
- 댓글 언급
- 프로젝트 변경

대시보드는 역할별로 다르게 보여준다.

- 일반 직원: 내 요청, 내 업무, 승인 대기, 기한 임박
- 업무 관리자: 미분류 업무, 지연 업무, 담당자 없는 업무
- 프로젝트 관리자: 프로젝트별 진행률, 차단 업무, 마일스톤 위험
- 경영진: 전사 업무량, 지연 현황, 프로젝트 현황, 조직별 병목

## AI 에이전트 확장 방향

AI는 1차 구현에서 제외하지만 다음 순서로 붙이는 것이 적절하다.

### 1단계: 읽기 전용 보조

- 번역 보조
- 업무 요약
- 프로젝트 진행 요약
- 지연 업무 설명
- 회의록에서 할 일 후보 추출
- 사내 자료 검색

### 2단계: 초안 작성

- 업무 설명 초안
- 보고서 초안
- 회의록 정리
- 완료 보고 초안
- 프로젝트 주간 보고 초안

### 3단계: 추천

- 담당자 후보 추천
- 우선순위 추천
- 유사 업무 추천
- 누락된 체크리스트 제안
- 지연 위험 감지

### 4단계: 제한적 자동화

- 정기 보고 생성
- 반복 업무 생성
- 승인자에게 리마인드
- 프로젝트 위험 알림

AI는 항상 사용자 권한 범위 안의 데이터만 읽어야 한다. AI가 업무 생성, 상태 변경, 승인, 종료 같은 실제 액션을 수행할 때는 사람의 확인을 받는 구조로 둔다.

## 배포와 보안

시스템은 회사 내부망 전용으로 운영한다.

권장 배포 구조는 다음과 같다.

- Docker Compose 기반
- Web/API 서버
- PostgreSQL
- 파일 저장 볼륨
- 내부망 reverse proxy
- 정기 백업

1차 보안 기준은 다음과 같다.

- 내부망 접속만 허용
- 비밀번호 해시 저장
- 세션 만료
- 계정 비활성화
- 역할 기반 접근 제어
- 업무 상태 변경 이력
- 관리자 감사 로그

2차 이후에는 비밀번호 정책, 로그인 실패 잠금, 2단계 인증, IP 제한, 중요 작업 재인증을 검토한다.

## 1차 출시 범위

1차 버전은 다음 기능을 포함한다.

- 로그인
- 직원 계정 관리
- 조직도 관리
- 기본 역할/권한
- 사용자 언어 설정
- 조직 기본 언어 설정
- 라이트/다크 테마
- 업무 항목 등록
- 빠른 업무 등록
- 업무 분류/승인/배정/진행/검토/종료
- 내 업무함
- 전사 업무 목록
- 검색과 필터
- 상태 보드
- 프로젝트 생성과 구성원 관리
- 프로젝트별 업무 연결
- 댓글
- 파일 첨부
- UI 다국어 리소스 구조
- 업무 콘텐츠 원문 언어 저장

- 회사 용어집
- 알림
- 기본 대시보드
- 업무 변경 이력

## 후속 출시 범위

2차 이후 기능은 다음 순서로 진행한다.

1. 승인/결재 고도화
2. 정기 업무와 템플릿
3. 보고서와 KPI
4. 문서/지식 관리
5. 번역 보조 에이전트 엔진 연결
6. AI 읽기 전용 보조
7. AI 초안 작성
8. AI 추천과 제한적 자동화
9. 자산, 구매, HR, 고객관리 등 업무 모듈 확장

## 성공 기준

1차 시스템이 성공했는지는 다음으로 판단한다.

- 주요 업무가 메신저나 구두 지시가 아니라 시스템에 등록된다.
- 담당자 없는 업무가 줄어든다.
- 기한 초과 업무가 보인다.
- 대표와 관리자가 업무 현황을 따로 물어보지 않아도 확인할 수 있다.
- 직원이 “내가 지금 해야 할 일”을 한 화면에서 볼 수 있다.
- 한국어 사용자와 영어 사용자가 같은 업무 항목을 각자 이해 가능한 언어로 확인할 수 있다.
- 완료와 반려의 근거가 이력으로 남는다.
- 반복 업무가 점차 템플릿화된다.

## 주요 위험과 대응

### 위험: 시스템이 너무 무거워져서 직원이 쓰지 않는다

대응: 처음 화면은 `내 업무`, `업무 등록`, `업무 목록`, `프로젝트` 중심으로 단순하게 만든다. 승인과 분류는 필요한 업무에만 적용한다.

### 위험: 대표 지시가 여전히 구두나 메신저에 남는다

대응: 대표나 비서, 업무 관리자가 빠르게 지시를 대신 등록할 수 있는 `빠른 등록` 기능을 둔다.

### 위험: 권한이 복잡해져 운영이 어렵다

대응: 내부 구조는 Role + Permission + Scope로 만들되, 1차 화면은 표준 역할만 제공한다.

### 위험: AI를 나중에 붙이기 어렵다

대응: 업무 항목, 댓글, 파일, 이력, 권한을 구조화해 저장한다. AI 전용 로그와 권한 검사를 후속 단계에서 추가할 수 있게 한다.

### 위험: 번역 때문에 업무 의미가 왜곡된다

대응: 원문을 항상 보존하고, 자동 번역 표시와 검토 상태를 둔다. 회사 용어집을 사용해 제품명, 고객명, 부서명, 내부 약어가 임의로 번역되지 않게 한다.

### 위험: 언어와 테마를 나중에 붙이기 어렵다

대응: 1차부터 UI 문구를 번역 리소스로 분리하고, 색상은 디자인 토큰으로 관리한다. 화면 컴포넌트는 긴 영어 문구와 한국어 문구가 모두 깨지지 않게 설계한다.

## 최종 판단

가장 적절한 1차 방향은 **전사 업무 허브형 Work OS**다.

이 방향은 현재 체계가 약한 회사에 맞다. ERP처럼 무겁지 않고, 단순 프로젝트 관리툴처럼 좁지도 않다. 회사의 일을 먼저 한곳에 모으고, 상태와 책임을 보이게 만든 뒤, 프로젝트, 결재, 문서, AI 에이전트를 단계적으로 붙일 수 있다.
